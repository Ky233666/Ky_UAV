# 本科毕设最终交付与答辩版本功能说明

> 编写时间：2026-04-02  
> 适用对象：`Ky_UAV` 当前版本的本科毕业设计最终交付与答辩准备  
> 适用范围：`Assets/Scenes/Main/MainScene.unity` 及当前 `main` 分支源码

## 1. 文档目的

这份文档用于明确本项目在“最终交付版本”和“毕业答辩版本”中，应该实现到什么程度，哪些功能必须具备，哪些功能属于加分项，哪些内容不建议在最后阶段继续扩张。

它不是单纯的功能罗列，而是给当前项目划定一个清晰的交付边界，避免后期继续无节制加功能，导致答辩前版本不稳定。

---

## 2. 最终交付版本的定位

本项目最终交付版本不应再被定义为“一个会飞的 Unity 场景”，而应被定义为：

一个能够完成多无人机任务分配、路径规划、障碍环境演示、运行时交互、结果展示与基础实验记录的仿真系统。

对本科毕设来说，最终答辩版本最重要的不是“算法做得多复杂”，而是以下四件事必须形成闭环：

1. 系统能稳定运行。
2. 算法能切换并展示差异。
3. 结果能被观察、说明和记录。
4. 打包后仍能完成主要交互，而不是只能在 Inspector 中调参。

---

## 3. 最终版本必须实现的功能

下面的功能清单，是基于当前源码状态得出的“最终交付答辩版最低标准”。

### 3.1 仿真基础运行能力

交付要求：

- 主场景能够稳定运行，不依赖临时手工修补。
- 系统支持 `开始仿真 / 暂停 / 重置`。
- 支持多架无人机统一生成、统一管理、统一重置。
- 打包后仍能完成基本操作，不依赖 Inspector 才能演示。

当前基础：

- `SimulationManager.cs` 已管理 `Idle / Running / Paused`。
- `DroneManager.cs` 已支持多机生成和运行时重建机群。
- 主界面底部已具备开始、暂停、重置控制。

最终判定标准：

- 在主场景中连续多次开始、暂停、重置后，机群状态仍然正确。
- 无人机数量调整后，机群能重新生成并继续参与仿真。

### 3.2 任务点系统与运行时交互

交付要求：

- 支持导入任务点。
- 支持运行时手动新增任务点。
- 支持清空任务点。
- 放置任务点时应具备基本合法性检查，避免放到建筑内部或不可达区域。

当前基础：

- `TaskPointUIManager.cs` 已支持点击放置、预览、`Esc` 取消。
- `TaskPointImporter.cs` 已支持 CSV / JSON 导入。
- `TaskPointSpawner.cs` 已支持避开障碍的安全生成。

最终判定标准：

- 答辩演示时，可以在运行中新增任务点并重新分配任务。
- 导入任务点后，系统能正常进入仿真流程。

### 3.3 多无人机任务调度能力

交付要求：

- 系统必须具备至少两种调度策略。
- 调度策略之间必须可以切换，并能解释差异。
- 调度结果要真实影响任务分配，而不是界面切换但行为不变。

当前基础：

- 已接入 `ISchedulerAlgorithm` 接口层。
- 已实现 `EvenSplitScheduler`。
- 已实现 `GreedyNearestScheduler`。
- 运行时面板已支持调度算法切换。

最终判定标准：

- 在相同任务点集合下，不同调度策略能够产生可观察的分配差异。
- 答辩时能说明“为什么使用该策略、该策略与对照组有何区别”。

### 3.4 路径规划与障碍环境能力

交付要求：

- 系统必须具备至少两种路径规划模式。
- 至少一种路径规划模式应能在障碍环境中绕障。
- 路径规划结果必须真实驱动无人机按 waypoint 飞行，而不是只飞直线终点。

当前基础：

- 已接入 `IPathPlanner` 接口层。
- 已实现 `StraightLinePlanner`。
- 已实现 `AStarPlanner`。
- `DroneStateMachine.cs` 已按路径点逐点执行。
- `DroneManager.cs` 已接入障碍层自动配置与建筑代理碰撞体。

最终判定标准：

- 在主场景中，A* 应能稳定展示相对直线规划的绕障行为。
- 答辩时能够明确说明“直线规划”和“A*”的差异与适用场景。

### 3.5 路径、轨迹与状态可视化

交付要求：

- 显示规划路径。
- 显示实际飞行轨迹。
- 能观察无人机当前执行状态。
- 能通过镜头切换直观看到任务执行过程。

当前基础：

- `DronePathVisualizer.cs` 已支持规划路径和轨迹显示。
- `CameraManager.cs` 已支持总览、跟随、多机切换。
- 运行时面板已支持路径与轨迹显示开关。

最终判定标准：

- 在答辩演示中，老师能够直观看到“规划结果”和“真实执行结果”。
- 视角切换后，仍可清晰观察单机与机群表现。

### 3.6 打包可用的运行时控制面板

交付要求：

- 将核心运行参数从 Inspector 迁移到 UI。
- 运行时面板应支持展开/收起，避免遮挡主要内容。
- 面板必须适用于打包演示，而不是只适合编辑器内部测试。

当前基础：

- `SimulationRuntimeControlPanel.cs` 已正式接入。
- 当前已支持：
  - 调度算法切换
  - 路径规划切换
  - 无人机数量调整
  - 飞行速度调整
  - 仿真倍速调整
  - 规划路径显示开关
  - 飞行轨迹显示开关
  - 总览 / 跟随 / 下一架镜头控制
  - 同步设置
  - 重建机群

最终判定标准：

- 答辩时核心演示流程不需要打开 Inspector。
- 打包版本中可直接切换算法、调整数量、切换视角和重建机群。

### 3.7 实时统计与结果展示

交付要求：

- 必须提供一个面向答辩和实验展示的统计区域。
- 至少显示当前算法、任务进度、机群数量、单机状态、总耗时等信息。
- 统计信息应来自真实运行数据，而不是写死文本。

当前基础：

- `DroneData.cs` 已有部分统计字段：
  - `completedTasks`
  - `totalFlightDistance`
  - `state`
  - `waitReason`
- 当前运行时面板只有摘要，不是正式统计面板。

仍需补齐：

- 总任务数 / 已完成任务数
- 各无人机状态
- 各无人机完成任务数
- 各无人机飞行距离
- 当前调度算法
- 当前路径规划算法
- 仿真总耗时

最终判定标准：

- 演示时不仅能看到无人机在飞，还能看到系统已经跑出了什么结果。
- 统计面板内容随仿真推进而实时更新。

### 3.8 实验结果导出能力

交付要求：

- 系统必须具备基础实验结果导出能力。
- 建议以 CSV 为主，必要时可附加 JSON。
- 导出的结果应能直接用于论文中的实验表格或结果整理。

当前基础：

- 当前项目已具备任务点导入的 CSV / JSON 解析能力。
- 但尚未具备正式的实验结果导出模块。

仍需补齐：

- 单次实验结果数据结构
- 仿真结束时的结果汇总
- CSV 导出逻辑
- 导出文件命名与保存路径约定

建议最少导出字段：

- 实验时间
- 无人机数量
- 任务点数量
- 调度算法
- 路径规划算法
- 总耗时
- 总完成任务数
- 总飞行距离
- 等待次数
- 备注

最终判定标准：

- 至少能在每次完整仿真结束后导出一条实验记录。
- 导出结果可直接用于论文表格整理。

### 3.9 基础多机协同与简单冲突规避

交付要求：

- 最终答辩版本不一定要做复杂协同算法，但至少要补一个“可解释的简单规避策略”。
- 该策略的目标不是追求论文级最优，而是让多机协同更完整、更可信。

当前基础：

- `DroneStateMachine.cs` 已有 `Waiting` 状态。
- `DroneData.cs` 已有 `waitReason` 字段。
- 当前状态机内部已明确预留冲突检测逻辑入口，但尚未真正实现。

建议最小实现方案：

- 同一路径段临时等待
或
- 近距离让行
或
- 局部暂停后恢复移动

最终判定标准：

- 当两机路径接近或交叉时，系统具备基本等待/让行行为。
- 答辩时可以明确说明：系统已考虑多机协同中的基础冲突处理。

## 4. 基于当前源码的完成度判断

### 4.1 当前已经基本具备的模块

- 仿真基础运行能力
- 任务点导入与运行时新增
- 多无人机统一管理
- 调度算法切换
- 路径规划算法切换
- 障碍环境路径规划基础能力
- 路径与轨迹可视化
- 相机系统
- 打包可用的运行时控制面板第一版

### 4.2 当前距离最终答辩版还差的关键模块

- 正式实时统计面板
- 实验结果导出
- 基础多机冲突规避
- 少量实验级参数的进一步 UI 化

### 4.3 真实判断

当前项目已经具备“可演示系统”的主体骨架，距离“本科毕设最终交付答辩版”并不远。真正还缺的不是重写整个系统，而是补上三块闭环：

1. 统计闭环
2. 导出闭环
3. 多机协同闭环

只要把这三块补齐，当前系统就已经足以支撑完整的毕业设计交付与答辩说明。

## 5. 最终答辩版本建议的功能优先级

### P0：必须完成

- 多无人机稳定运行
- 任务点导入 / 新增 / 清空
- 调度算法双方案切换
- 路径规划双方案切换
- A* 绕障演示
- 路径与轨迹可视化
- 打包可用运行时控制面板
- 实时统计面板
- 实验结果导出
- 一种简单多机冲突规避策略

### P1：建议完成

- 更多运行时参数迁移到 UI
- 更清晰的单机状态展示
- 更好的实验记录命名与归档方式
- 更稳定的任务集切换入口

### P2：有余力再做

- 第二种更强的调度算法
- 第二种更强的路径规划算法
- 批量实验
- 图表面板
- 能耗模型
- 动态障碍物

## 6. 最终答辩演示建议流程

建议答辩时按以下顺序演示：

1. 展示主界面和运行时控制面板，说明系统可打包交互。
2. 导入或新增任务点，说明系统支持运行时任务配置。
3. 选择第一种调度算法和第一种路径规划算法，开始仿真。
4. 切换到第二种调度或路径规划算法，展示差异。
5. 切换总览与跟随镜头，展示单机与机群执行过程。
6. 展示规划路径、实际轨迹和统计面板。
7. 仿真结束后导出一份实验记录。
8. 最后总结系统已完成“任务配置、任务分配、路径规划、轨迹展示、结果统计、结果导出”的完整闭环。

---

## 7. 最后阶段不建议继续扩大的范围

在答辩前最后阶段，不建议继续把时间投入到下面这些方向：

- 大幅重做 UI 视觉风格
- 引入过多新算法导致系统不稳定
- 追求复杂三维飞行动力学
- 引入动态障碍物并重新改写规划框架
- 做过重的论文级优化而忽略交付稳定性

最后阶段最重要的是：

- 版本稳定
- 功能闭环
- 演示顺畅
- 数据可说明

---

## 8. 结论

结合当前源码状态，本项目的本科毕设最终交付答辩版本，应该至少实现以下能力：

- 多无人机任务执行
- 调度算法对比
- 路径规划算法对比
- 障碍环境绕障
- 路径与轨迹可视化
- 运行时任务交互
- 打包可用运行时控制面板
- 实时统计展示
- 实验结果导出
- 一种简单可解释的多机冲突规避策略

如果以上功能全部完成，那么这个项目就不再只是“一个演示场景”，而可以被明确地定义为：

一个具备算法对比、结果展示与实验支撑能力的本科毕业设计仿真系统。
