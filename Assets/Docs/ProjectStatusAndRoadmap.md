# 项目进展与后续开发路线

## 1. 文档用途

这份文档是当前项目的唯一总说明文档，用来回答三件事：

1. 这个项目现在已经做到了什么程度。
2. 这个项目距离“可用于本科毕设展示与论文撰写”还差哪些关键内容。
3. 后续继续开发时，应该优先补哪些功能，以及每个功能做到什么程度算完成。

后续如果需要继续开发本项目，默认先阅读本文档，再决定下一步工作。  
旧的阶段性说明、教程型文档和历史进展报告已经删除，避免信息重复和状态失真。

## 2. 项目定位

### 2.1 课题名称

基于 Unity 的多无人机协同路径规划与调度算法仿真系统设计与实现

### 2.2 项目目标

本项目最终要做成一个可以用于毕设演示、论文实验和结果对比的仿真系统。  
它不只是“让几个无人机飞起来”，而是要具备下面这些核心能力：

1. 支持多无人机执行多任务点巡检。
2. 支持路径规划算法接入与切换。
3. 支持任务调度算法接入与切换。
4. 支持基础障碍环境下的路径规划与避障展示。
5. 支持轨迹、状态、统计指标的可视化。
6. 支持算法实验结果对比与数据导出。

### 2.3 当前真实定位

当前项目更接近于：

- 一个已经具备基础场景、基础 UI、任务点系统、多无人机生成和基础飞行逻辑的仿真原型。

当前项目还不是：

- 一个已经完成“多无人机协同路径规划与调度算法对比实验”的完整毕设系统。

一句话概括：

当前已经完成了“平台骨架”和“演示原型”，还没有完成“算法核心”和“实验系统”。

## 3. 当前项目结构

### 3.1 运行环境

- Unity 版本：`2022.3.62f3`
- 开发语言：`C#`
- 当前主场景：`Assets/Scenes/Main/MainScene.unity`
- 当前核心脚本目录：`Assets/Scripts/UAV/Controller`

### 3.2 核心脚本现状

当前主要脚本及其职责如下：

- `SimulationManager.cs`
  - 管理仿真状态：`Idle / Running / Paused`
  - 绑定开始、暂停、重置按钮
  - 启动时调用 `DroneManager.AutoAssignTasks`

- `DroneManager.cs`
  - 负责批量生成无人机
  - 维护无人机列表和 `DroneData`
  - 自动为每架无人机关联 `DroneStateMachine`
  - 当前任务分配方式是简单均分，不是优化调度算法

- `DroneController.cs`
  - 负责无人机移动控制
  - 当前运动方式是匀速直线飞向目标
  - 若存在状态机，则由状态机主导移动逻辑

- `DroneStateMachine.cs`
  - 当前状态：`Idle / Moving / Waiting / Finished`
  - 支持按任务队列依次飞行
  - `Waiting` 仅为预留状态，真正的冲突检测与避障还未实现

- `DroneData.cs`
  - 保存无人机编号、任务队列、任务索引、完成任务数、等待原因等信息
  - 已具备做统计面板的基础数据结构，但没有完整使用起来

- `TaskPoint.cs`
  - 管理任务点状态：`Pending / InProgress / Completed`
  - 支持颜色可视化
  - 目前任务完成状态更新并没有形成完整的调度闭环

- `TaskManager.cs`
  - 维护场景中任务点列表
  - 支持按优先级排序和获取待执行任务
  - 目前没有成为整个系统的统一调度入口

- `TaskPointSpawner.cs`
  - 支持运行时动态创建任务点

- `TaskPointImporter.cs`
  - 支持从 `Resources` 导入 CSV / JSON 任务点数据

- `TaskPointUIManager.cs`
  - 管理新增任务点、清空任务点、导入任务点的 UI 按钮

- `CameraManager.cs`
  - 支持总览视角和跟随视角切换

- `SimulationDashboardStyler.cs`
  - 负责当前主界面“指挥台风格”UI 的自动样式化

### 3.3 当前系统运行逻辑

当前运行流程大致如下：

1. 进入 `MainScene`
2. `DroneManager` 自动生成多架无人机
3. 任务点通过场景预放置、运行时创建或文件导入产生
4. 点击“开始仿真”
5. `SimulationManager` 收集任务点并调用 `DroneManager.AutoAssignTasks`
6. 各无人机状态机进入 `Moving`
7. 无人机按分到的任务顺序直线飞向任务点
8. 到达后继续切换到下一个任务，直到任务队列结束

这个流程可以跑通，但它本质上还是“顺序执行任务队列”，还不能称为路径规划与调度算法系统。

## 5. 当前没有完成但毕设必须补上的内容

下面这些是本项目距离“合格毕设系统”最关键的缺口。

### 5.1 算法接口层

当前问题：

- 还没有抽象出统一的路径规划接口和调度接口
- 算法代码未来如果直接写进现有控制器，会导致耦合很重
- 运行时算法切换目标目前没有架构支撑

必须补：

- `IPathPlanner`
- `ISchedulerAlgorithm`
- 算法输入输出数据结构
- 算法选择配置入口

这是后续所有算法工作的前置条件。

### 5.2 路径规划算法

当前问题：

- 无人机当前是直线飞行，不是规划路径
- 没有网格、导航图、采样树或 waypoint 路径
- 没有障碍建模后的寻路逻辑

必须补：

- 至少实现一种可演示、可说明的路径规划算法
- 最推荐先做 `A*`
- 如果时间允许，再补 `RRT` 或 `RRT*` 作为对比组

最低要求：

- 给定起点、终点、障碍物后，能够生成一条非直线、可避障的路径
- 路径可被无人机沿 waypoint 执行

### 5.3 调度算法

当前问题：

- 当前只是把任务点平均切分给无人机
- 没有基于距离、代价、优先级、时间窗等条件进行优化

必须补：

- 至少实现一种任务调度算法
- 最推荐先做“最近任务优先”的贪心调度
- 如果时间允许，再加一种更复杂的对比算法

最低要求：

- 不同无人机对任务点的分配结果不再是简单均分
- 能基于某个明确规则生成任务分配方案

### 5.4 避障与冲突处理

当前问题：

- `Waiting` 状态只是占位
- 没有多机冲突检测
- 没有静态障碍与动态障碍的统一处理

必须补：

- 静态障碍物检测
- 多无人机最基础的冲突规避机制
- 最低限度的等待或让行策略

最低要求：

- 无人机不会直接穿过建筑
- 两架无人机不会长期占据同一路径点造成死锁

### 5.5 指标统计与实验输出

当前问题：

- 虽然 `DroneData` 里已经有统计字段，但没有完整更新和展示
- 没有统一记录总飞行距离、完成任务数、总耗时、等待次数、冲突次数
- 没有实验数据导出

必须补：

- 实时指标面板
- 结果汇总结构
- CSV 导出

最低要求：

- 能导出一组实验数据，用于论文表格和图表整理

## 6. 当前最真实的开发判断

### 6.1 可以认为已经完成的层

- 场景基础层
- UI 初版层
- 任务点基础管理层
- 多无人机生成与基础控制层

### 6.2 只能算“已搭骨架”的层

### 6.3 目前最欠缺的不是界面，而是算法和实验链路

UI 已经可以继续迭代，但它不是当前最影响毕设完成度的地方。  
当前最影响项目是否成立的，是以下三件事：

1. 没有真正的路径规划算法。
2. 没有真正的调度算法。
3. 没有实验指标与结果导出。

如果这三块不补上，项目只能算“无人机演示系统”，还不能支撑“多无人机协同路径规划与调度算法仿真系统”这个题目。

## 7. 后续开发总原则

后续继续开发时，必须遵守下面的顺序，避免越做越乱。

### 原则 1：先补接口，再补算法，再补展示

正确顺序：

1. 抽象接口
2. 接入最小可用算法
3. 跑通完整流程
4. 补 UI、可视化、统计和导出

不要先做：

- 复杂界面
- 花哨特效
- 论文格式包装

在算法未落地前，这些都不是主线。

### 原则 2：先做“最小可交付版本”，不要一开始追求最强算法

建议先完成：

- 贪心调度
- A* 路径规划
- 简单障碍建模
- 基础轨迹显示
- 基础统计导出

这套组合已经足够形成一个能答辩、能写实验章节、能展示对比思路的毕设最小闭环。

### 原则 3：每补一个模块，都要明确是否服务于论文实验

每个功能都要问自己：

- 它能不能帮助我形成实验数据？
- 它能不能帮助我说明算法效果？
- 它能不能帮助我写论文中的系统设计或实验分析章节？

如果不能，优先级就应该往后放。

## 8. 推荐开发路线

这一部分是后续继续开发时的默认执行顺序。

### P0：必须先完成

#### 任务 1：抽象算法接口

目标：

- 把路径规划和任务调度从 `DroneManager` / `SimulationManager` 中解耦出来

建议产出：

- `IPathPlanner`
- `ISchedulerAlgorithm`
- `PathPlanningRequest`
- `PathPlanningResult`
- `SchedulingRequest`
- `SchedulingResult`

完成标准：

- 不改业务主流程也能切换算法实现类

#### 任务 2：实现最小可用调度算法

推荐先做：

- 贪心调度

建议规则：

- 最近任务优先
- 可选加入优先级权重

完成标准：

- 输入多架无人机和一组任务点
- 输出每架无人机各自的任务队列
- 结果不再是简单平均分配

#### 任务 3：实现最小可用路径规划算法

推荐先做：

- A*

建议目标：

- 先在二维平面或简化网格上跑通
- 再映射到 Unity 场景位置

完成标准：

- 无人机不是直接飞向目标
- 而是沿规划路径点移动
- 能绕开障碍物到达任务点

### P1：完成毕设展示闭环

#### 任务 4：补障碍建模与避障

建议内容：

- 给建筑或障碍体设置统一层
- 路径规划时把障碍纳入可达性判断
- 对多机冲突先采用简单策略，例如等待或重规划

完成标准：

- 静态障碍可避让
- 多机不会明显互穿或长期卡死

#### 任务 5：补轨迹可视化

建议内容：

- `LineRenderer` 显示已飞轨迹
- 可选显示规划路径
- 可选显示任务连接顺序

完成标准：

- 运行时能直观看出不同算法的路径差异

#### 任务 6：补实时统计面板

建议指标：

- 总任务数
- 已完成任务数
- 单机完成任务数
- 单机飞行距离
- 总耗时
- 等待次数
- 冲突次数

完成标准：

- Unity 界面中可实时看到关键指标

### P2：形成论文实验能力

#### 任务 7：增加第二种算法做对比

建议优先级：

1. 第二种调度算法
2. 第二种路径规划算法

可选组合：

- 调度：贪心 vs 遗传算法
- 路径规划：A* vs RRT

完成标准：

- 至少能做一组算法对比实验

#### 任务 8：增加实验参数面板

建议参数：

- 无人机数量
- 任务点数量
- 障碍物数量或密度
- 规划算法选择
- 调度算法选择
- 飞行速度

完成标准：

- 不改代码即可切换核心实验配置

#### 任务 9：导出实验结果

建议格式：

- CSV

建议字段：

- 实验编号
- 时间戳
- 无人机数量
- 任务点数量
- 调度算法名
- 路径规划算法名
- 总耗时
- 总飞行距离
- 平均单机任务数
- 冲突次数
- 成功率

完成标准：

- 可以直接把导出数据拿去做论文表格和图表

### P3：有余力再做

可选增强项：

- 更真实的飞行运动学
- 任务时间窗
- 电量或能耗模型
- 动态障碍物
- 三维路径规划
- 更复杂的 UI 数据看板
- 实验批量运行

这些内容有价值，但不应阻塞 P0 和 P1。

## 9. 建议的毕设最小可交付版本

如果时间有限，建议把目标控制在下面这条线上：

1. 多无人机生成与任务点导入可正常运行。
2. 实现贪心任务调度。
3. 实现 A* 路径规划。
4. 让无人机沿规划路径飞行并避开静态障碍。
5. 在 UI 中显示基础运行指标。
6. 支持至少两组实验配置。
7. 支持导出实验结果 CSV。

只要这条线完成，项目就已经具备较完整的毕设交付能力。

---

## 10. 当前建议不要优先投入的方向

在以下内容上不要过早投入大量时间：

- 过度复杂的 UI 动效
- 过度真实的物理飞行模型
- 一开始就做遗传算法、蚁群算法等复杂实现
- 批量重构整个项目目录结构
- 非必要的多场景系统化封装

原因很简单：

当前项目最缺的是“题目成立所必须的功能”，不是“额外锦上添花的功能”。

## 11. 后续继续开发时的默认执行清单

如果以后继续让 AI 或协作者补项目，默认按下面顺序推进：

1. 先阅读本文件。
2. 优先检查 P0 是否都未完成。
3. 如果 P0 未完成，禁止优先做美化类工作。
4. 每完成一个功能，都要更新本文件中的“完成情况”和“下一步建议”。
5. 每次新增算法，都要说明：
   - 算法解决什么问题
   - 输入输出是什么
   - 和现有系统如何集成
   - 如何验证它确实生效

## 12. 当前结论

当前项目已经完成了一个不错的 Unity 仿真原型，但距离“多无人机协同路径规划与调度算法仿真系统”这个题目，还有关键的算法层和实验层工作没有做。

当前最合理的策略不是继续横向扩展，而是集中力量补三条主线：

1. 算法接口与架构解耦
2. 贪心调度 + A* 路径规划的最小闭环
3. 指标统计、结果展示与实验导出

只要先把这三条主线补齐，这个项目就能从“原型”真正升级为“可用于毕设答辩和论文分析的系统”。
