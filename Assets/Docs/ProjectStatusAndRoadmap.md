# 项目进展与后续开发路线

## 1. 文档用途

这份文档是当前项目的统一总说明，用来解决四件事：

1. 当前项目已经做到什么程度。
2. 哪些功能已经真实完成，哪些还只是骨架。
3. 现在最适合继续补什么。
4. 后续如果新开线程或交给新的协作者，应该按什么顺序推进。

以后继续开发本项目时，默认先阅读本文件，再决定下一步工作。

---

## 2. 项目定位

### 2.1 课题名称

基于 Unity 的多无人机协同路径规划与调度算法仿真系统设计与实现

### 2.2 项目目标

本项目最终目标不是单纯“让几架无人机飞起来”，而是形成一个可用于本科毕设展示、论文实验和结果对比的仿真系统。系统应具备以下能力：

1. 支持多无人机执行多任务点巡检。
2. 支持任务调度算法切换与对比。
3. 支持路径规划算法切换与对比。
4. 支持静态障碍环境下的路径规划与避障演示。
5. 支持路径、轨迹、状态、统计指标的可视化。
6. 支持实验结果汇总与导出。

### 2.3 当前真实定位

截至当前版本，项目已经从“基础原型”进入“最小可用算法演示版”阶段。

更准确地说：

- 平台骨架已经完成。
- 第一版调度算法已经接入。
- 第一版路径规划算法已经接入。
- 无人机已经可以按规划路径点飞行，而不是只会直线冲向目标。
- 路径和飞行轨迹已经可以直接可视化。
- 任务点交互、视角交互、基础避障交互已经具备演示价值。

但它还不是完整毕设成品，因为以下关键部分仍未完成：

- 障碍环境的正式实验闭环
- 多机冲突规避
- 实时统计面板
- 实验结果导出
- 第二组更强的对比算法

一句话总结：

当前项目已经不是单纯场景演示，而是已经具备“算法接入、任务分配、路径执行、可视化观察”闭环的第一版仿真系统。

## 3. 当前系统结构

### 3.1 运行环境

- Unity 版本：`2022.3.62f3`
- 开发语言：`C#`
- 当前主场景：`Assets/Scenes/Main/MainScene.unity`
- 当前主要脚本目录：
  - `Assets/Scripts/UAV/Controller`
  - `Assets/Scripts/UAV/Comm`
  - `Assets/Scripts/UAV/Model`

### 3.2 核心模块概览

- `SimulationManager.cs`
  - 管理仿真状态：`Idle / Running / Paused`
  - 绑定开始、暂停、重置按钮
  - 启动时收集任务点并触发任务分配
  - 当场景没有任务点时可自动导入默认任务点数据

- `DroneManager.cs`
  - 统一生成、管理和重置无人机
  - 调用调度算法分配任务
  - 调用路径规划算法生成路径
  - 维护无人机数据、状态机、路径可视化组件
  - 支持 `SpawnPoint` 起飞点
  - 支持障碍层自动配置与建筑碰撞代理处理

- `DroneController.cs`
  - 负责无人机基础运动控制
  - 当前采用脚本驱动位移，不依赖刚体推进
  - 已修复 `Play` 后乱飞问题

- `DroneStateMachine.cs`
  - 管理 `Idle / Moving / Waiting / Finished`
  - 按任务队列执行任务
  - 已支持沿 waypoint 路径逐点飞行

- `DroneData.cs`
  - 存储编号、任务队列、状态、完成数等信息
  - 已存储路径规划相关字段：
    - `plannedPath`
    - `currentWaypointIndex`
    - `currentPlannerName`

- `TaskManager.cs`
  - 管理场景任务点集合
  - 获取待执行任务并进行排序

- `TaskPoint.cs`
  - 管理任务点状态：`Pending / InProgress / Completed`
  - 用颜色体现任务状态

- `TaskPointSpawner.cs`
  - 运行时创建任务点
  - 支持避开建筑和障碍的安全生成

- `TaskPointImporter.cs`
  - 从 `Resources` 导入 CSV / JSON 任务点

- `TaskPointUIManager.cs`
  - 负责手动添加任务点交互
  - 已支持点击场景放置任务点
  - 已支持落点合法性检查与预览圈

- `CameraManager.cs`
  - 管理总览视角与跟随视角
  - 总览视角已支持自由移动
  - 跟随视角已支持切换目标无人机

- `SimulationDashboardStyler.cs`
  - 管理主界面风格统一
  - 负责标题卡、底部控制区等 UI 样式整理
  - 已修复编辑器阶段 `OnValidate` 触发的警告

- `DronePathVisualizer.cs`
  - 显示规划路径
  - 显示实际飞行轨迹

- `IPathPlanner.cs / ISchedulerAlgorithm.cs`
  - 算法接口层
  - 负责让算法实现与控制层解耦

- `AStarPlanner.cs / StraightLinePlanner.cs`
  - 路径规划算法实现

- `EvenSplitScheduler.cs / GreedyNearestScheduler.cs`
  - 调度算法实现

## 4. 当前已经完成的内容

## 4.1 仿真平台基础骨架

已完成：

- 主场景、地面环境、建筑、无人机、任务点、相机、UI 的基础搭建
- 开始、暂停、重置的仿真控制链路
- 多无人机统一生成与统一管理
- 主界面基础样式优化

当前结论：

- 已达到“可以稳定运行并进行演示”的程度
- 不再是只有静态场景和零散脚本的半成品

## 4.2 多无人机运行与稳定性修复

已完成：

- 统一由 `DroneManager` 生成受控无人机
- 清理了主场景中遗留的旧版单机对象问题
- 修复了“显示 5 架但只有 4 架受控”的问题
- 修复了 `Play` 后因刚体物理导致的乱飞问题
- 通过 `SpawnPoint` 让无人机在预设安全位置起飞
- 开始、暂停、重置目前能稳定控制受管无人机

当前结论：

- 多机基础运行闭环已经稳定
- 当前 5 架无人机可以统一响应仿真控制

## 4.3 任务系统与任务点交互

已完成：

- 场景内任务点识别
- 运行时新增任务点
- 清空任务点
- CSV / JSON 导入任务点
- 场景中没有任务点时自动导入默认任务点数据
- “新增任务点”已改为点击场景放置，不再随机刷新
- 放置模式支持 `Esc` 取消
- 放置模式支持鼠标落点预览圈
- 预览圈会区分可放置与不可放置状态
- 手动放置任务点时已阻止放进楼体或障碍区域
- 任务点 prefab 中心位置已与预览圈中心对齐
- 运行时随机/导入生成时支持安全重定位，避免任务点落在建筑内部

当前结论：

- 任务系统已经不只是“数据导入”，而是具备了较强的现场交互演示能力

当前限制：

- 缺少任务集保存/切换能力
- 缺少批量生成规则
- 还没有专门的实验任务集管理面板

## 4.4 算法接口层

已完成：

- `IPathPlanner`
- `ISchedulerAlgorithm`
- `PathPlanningRequest / PathPlanningResult`
- `SchedulingRequest / SchedulingResult`
- 调度算法类型枚举
- 路径规划算法类型枚举

当前结论：

- 算法层已经和控制层初步解耦
- 后续新增算法时，不需要把逻辑硬写回 `DroneManager`

这部分可以视为 `P0 / 任务 1` 已完成。

## 4.5 调度算法

已完成：

- `EvenSplitScheduler`
  - 作为基础对照组
  - 将任务点按数量均分给各无人机

- `GreedyNearestScheduler`
  - 作为当前第一版真实调度算法
  - 基于“最近优先”，并结合任务优先级权重

当前结论：

- 任务分配已经不再只是固定均分
- 已具备最基础的“调度算法可切换、可对比”能力

当前不足：

- 缺少第二种更复杂的调度算法
- 缺少调度层统计输出

这部分可以视为 `P0 / 任务 2` 已完成。

## 4.6 路径规划算法

已完成：

- `StraightLinePlanner`
  - 作为保底直线规划器

- `AStarPlanner`
  - 当前第一版真实路径规划算法
  - 基于 `XZ` 平面的二维网格搜索
  - 支持四方向 / 八方向搜索
  - 支持基于障碍层的阻挡检测
  - 支持 waypoint 路径输出
  - 已做过一轮防止拐角穿障与错误路径简化的修正

当前结论：

- 项目已经具备“路径规划算法切换”能力
- 已经不再是所有无人机都只会直线飞向目标

当前不足：

- 还缺少第二种路径规划算法用于对比
- A* 的避障效果目前已接入，但演示稳定性和表现力仍需继续加强

这部分可以视为 `P0 / 任务 3` 已基本完成。

## 4.7 路径执行与可视化

已完成：

- 无人机按 `plannedPath` 的 waypoint 逐点飞行
- 不再只是拿终点做直线移动
- 青蓝色规划路径显示
- 橙色实际飞行轨迹显示

当前结论：

- 已经可以在场景中直接观察“规划路径”和“真实飞行轨迹”
- 这是当前系统最有演示价值的部分之一

当前不足：

- 还没有显示搜索网格、障碍判定结果、路径长度等更细节的调试信息
- 还没有统一的运行时信息面板来显示当前算法和路径指标

## 4.8 相机与观察视角

已完成：

- `1` 键切换总览视角
- `2` 键切换跟随视角
- `Tab / E` 切换到下一架无人机
- `Q` 切换到上一架无人机
- 跟随相机平滑跟随
- 总览视角支持自由移动：
  - `W / A / S / D` 平移
  - `R / F` 升降
  - 鼠标滚轮推进缩放
  - 右键旋转
  - `Shift` 加速

当前结论：

- 当前视角系统已经能支持演示、调试和答辩展示

## 4.9 UI 与交互

已完成：

- 左上标题信息卡
- 底部仿真控制区
- 底部任务工具区
- 状态文本
- UI 整体风格优化
- 修复 `SimulationDashboardStyler` 在编辑器阶段触发的 `SendMessage cannot be called during Awake/OnValidate` 警告
- 已清理一轮右侧实验性残留侧栏，避免旧 UI 与当前 UI 重叠

当前不足：

- 当前 UI 还没有形成最终的“打包版运行时控制台”

## 6. 当前没有完成但毕设必须补上的内容

## 6.1 静态障碍环境闭环

虽然 `AStarPlanner` 已经接入障碍层检测，但这部分还没有形成足够稳定的实验闭环。

必须补上：

- 统一场景障碍物层
- 统一建筑碰撞体与规划边界
- 调整 `planningWorldMin / planningWorldMax / planningGridCellSize`
- 设计一组“必须绕障才能到达”的任务点布局
- 让 `StraightLine` 与 `AStar` 差异稳定可见

最低要求：

- 在主场景中可稳定复现绕障效果
- 不需要靠偶然位置才能看出 A* 的优势

## 6.2 多机冲突规避

当前 `Waiting` 状态仍主要是预留状态，尚未形成多机协同策略。

必须补上至少一种简单可解释的策略，例如：

- 同路径点等待
- 近距离让行
- 局部重规划

最低要求：

- 两架无人机不会长时间占用同一路段导致明显卡死

## 6.3 实时统计面板

当前 `DroneData` 已经有部分统计字段，但没有统一展示。

必须补上：

- 当前调度算法名
- 当前路径规划算法名
- 总任务数
- 已完成任务数
- 每架无人机完成任务数
- 每架无人机飞行距离
- 仿真总耗时
- 等待次数
- 冲突次数

最低要求：

- 在 Unity 界面中实时看到关键指标变化

## 6.4 实验数据导出

这是论文非常关键的一环，目前还没开始。

必须补上：

- 实验结果汇总结构
- CSV 导出
- 一次仿真结束后的结果落盘

最低要求：

- 至少能导出一行完整实验记录，供论文表格和图表使用

## 6.5 第二组对比算法

当前已有：

- 调度：`EvenSplit` vs `GreedyNearest`
- 路径规划：`StraightLine` vs `AStar`

这已经足够做基础对照，但说服力还不够强。

建议后续至少再补一项：

- 第二种调度算法
  或
- 第二种路径规划算法

优先建议：

- 路径规划补简化版 `RRT`
  或
- 调度补一个带代价函数的增强贪心/简化遗传算法

## 8. 当前建议的下一步

如果继续按毕设主线推进，下一步最推荐做的是：

1. 先把障碍层、碰撞体、规划范围和任务点布局调到稳定，确保 A* 绕障效果清晰可重复。
2. 再做一个实时统计面板，把当前算法名、任务进度、无人机状态和飞行距离显示到 UI。
3. 再把部分 Inspector 参数迁移到运行时 UI，提升打包后的交互性。
4. 最后再补实验数据导出。

当前不建议优先做的事情：

- 大量时间继续纯美化 UI
- 一上来就实现特别复杂的高级算法
- 大范围重构目录结构

原因很简单：

当前最缺的是“实验闭环”和“打包后交互闭环”，不是“界面更炫”。

## 9. 毕设最小可交付版本判断

如果能把下面几项做完，就已经具备比较完整的本科毕设交付能力：

1. 多无人机正常运行，任务点可手动放置、可导入，仿真可开始/暂停/重置。
2. 具备至少两种调度策略对照。
3. 具备至少两种路径规划策略对照。
4. A* 在障碍环境中可稳定演示绕障。
5. 无人机按规划路径飞行，并显示规划路径和实际轨迹。
6. UI 中可实时显示核心统计指标。
7. 可导出实验结果 CSV。

截至当前版本：

- 前 1 到 5 项的大部分骨架已经完成
- 第 4 项还需要场景和参数层继续完善
- 第 6 项和第 7 项还没有真正完成

## 10. 后续协作默认规则
