## 2. 项目定位

### 2.1 课题定位

本项目是一个基于 Unity 的多无人机协同巡检与路径规划仿真系统，目标不是单纯做一个“会飞的场景”，而是搭建一个可用于：

- 毕业设计展示
- 调度算法接入与对比
- 路径规划算法接入与对比
- 障碍环境下的路径可视化观察
- 多机任务执行演示
- 打包后的交互式展示

的完整仿真基础平台。

### 2.2 当前真实阶段

截至当前源码版本，项目已经从“原型场景”进入“最小可用演示系统”阶段，并且相较于早期版本，已经补齐了部分面向打包展示的运行时交互能力。

当前已经具备：

- 多无人机统一生成、统一管理、统一重置
- 调度算法切换与任务分配
- 路径规划算法切换与路径执行
- 规划路径与实际飞行轨迹可视化
- 任务点运行时添加、清空、导入
- 总览/跟随视角与多机观察
- 面向打包版的运行时控制面板

因此，当前版本已经不是“只能在 Inspector 里调参的内部原型”，而是具备较强答辩展示能力的第一版成型系统。

## 4. 当前系统结构

### 4.1 仿真总控

`SimulationManager.cs`

职责：

- 管理仿真状态：`Idle / Running / Paused`
- 绑定开始、暂停、重置按钮
- 在启动仿真时收集任务点并调用 `DroneManager` 进行任务分配
- 当场景中没有任务点时可自动导入默认任务点
- 运行时自动挂载 `SimulationRuntimeControlPanel`

现状判断：

- 已稳定可用
- 已成为当前场景运行流程的主入口

### 4.2 无人机管理

`DroneManager.cs`

职责：

- 统一生成和管理多架无人机
- 管理无人机数据、状态机和路径可视化组件
- 调用调度算法和路径规划算法
- 自动配置障碍层与建筑代理碰撞体
- 支持运行时重建机群、统一速度设置、统一轨迹可见性设置

当前支持的关键能力：

- `SpawnPoint` 起飞点优先生成
- 手动放置起飞点后按放置顺序生成机群
- 手动移动已有起飞点并保持生成顺序不变
- 手动删除单个起飞点与清空全部起飞点
- 起飞点顺序编号显示
- `RespawnDrones(int count)`
- `ApplyDroneSpeedToAll(float newSpeed)`
- `ApplyPathVisibilityToAll(bool showPlannedPath, bool showTrail)`
- 调度算法枚举切换：`EvenSplit / GreedyNearest`
- 路径规划枚举切换：`StraightLine / AStar`
- 障碍层自动配置
- 建筑根节点自动递归设层
- 建筑代理 `BoxCollider` 自动生成

现状判断：

- 这是当前项目最核心的控制层之一
- 已经具备“运行时交互驱动”的能力，不再只是 Inspector 配置层

### 4.3 单机控制与状态执行

`DroneController.cs`

职责：

- 管理无人机基础运动
- 支持目标点与虚拟目标位置两种模式
- 支持回到初始位置与重置

`DroneStateMachine.cs`

职责：

- 管理任务执行状态流转
- 按任务队列推进任务
- 驱动无人机按 waypoint 路径逐点飞行

现状判断：

- 基础执行链路已经建立
- 已补上第一版简单冲突规避策略
- 当前仍偏演示型，尚不属于更强的正式协同规避方案
- 经当前场景实测，在起飞区聚集、近距离交汇和目标点密集区域，仍可能出现机体碰撞重叠或视觉穿透；该问题初步判断更适合后续结合论文方案，按更系统的协同规避算法单独处理

### 4.4 路径可视化

`DronePathVisualizer.cs`

职责：

- 绘制当前规划路径
- 绘制实际飞行轨迹
- 支持分别开关规划路径和轨迹显示
- 在重置时清空并重新初始化轨迹

现状判断：

- 可视化效果已经能支撑课堂展示和算法对比
- 运行时面板已接入显示开关

### 4.5 任务点系统

`TaskPointUIManager.cs`

职责：

- 管理新增任务点、导入任务、清空任务按钮
- 支持点击场景放置任务点
- 支持落点合法性检测
- 支持鼠标落点预览圈
- 支持 `Esc` 取消放置模式
- 放置时自动切换到总览视角，便于交互

`TaskPointSpawner.cs`

职责：

- 运行时创建任务点
- 基于障碍物检测自动避开建筑和阻挡区域
- 支持不安全位置重试定位

`TaskPointImporter.cs`

职责：

- 从资源中导入任务点数据

现状判断：

- 当前任务点交互已比较完整
- 对答辩演示来说，交互性明显高于纯静态导入方案

### 4.6 相机系统

`CameraManager.cs`

职责：

- 管理总览相机与跟随相机
- 在多机之间切换跟随目标
- 支持总览相机自由移动、旋转和缩放

当前键位：

- `1`：总览视角
- `2`：跟随视角
- `Tab / E`：下一架无人机
- `Q`：上一架无人机
- `W/A/S/D`：总览平移
- `R / F`：升降
- 鼠标滚轮：缩放
- 右键拖动：旋转
- `Shift`：加速

现状判断：

- 已满足演示、观察和调试需求

### 4.7 主界面样式与运行时控制面板

`SimulationDashboardStyler.cs`

职责：

- 统一主界面的标题卡、状态卡、底部控制区和任务区样式
- 清理旧版残留右侧栏，避免与新面板重叠

`SimulationRuntimeControlPanel.cs`

职责：

- 在运行时自动创建右侧控制面板
- 支持展开/收起
- 支持滚动区域
- 支持摘要、状态提示和交互控制
- 使用更紧凑的布局，避免压住底部任务工具区

当前已接入的运行时功能：

- 调度算法切换
- 路径规划算法切换
- 路径规划网格尺寸调整
- 路径规划 X/Z 边界调整
- 对角搜索开关
- 障碍自动配置开关
- 无人机数量调整
- 飞行速度调整
- 仿真倍速调整
- 规划路径显示开关
- 飞行轨迹显示开关
- 跟随镜头高度与距离调整
- 实时统计信息卡
- CSV 实验结果导出
- 起飞点新增 / 移动 / 删除 / 清空交互
- 总览 / 跟随 / 下一架镜头控制
- 同步当前机群设置
- 重建机群

规划区参数说明：

- `网格` 对应 A* 的搜索精度，值越小越细，计算量也越大
- `X/Z 边界` 对应 A* 允许搜索的矩形范围
- `对角` 对应 A* 是否允许斜向扩展
- `障碍` 对应是否自动把场景建筑纳入规划障碍

现状判断：

- 右侧运行时面板已正式落地
- 它已经替代了部分过去只能在 Inspector 中修改的关键参数
- 这是当前版本相对于旧版最重要的一次交互升级

### 4.8 主场景与场景资源策略

当前情况：

- 主场景已经从早期手工拼装的简易楼体测试布局，切换为基于 `SimpleCityPackage` 的现成城市街区
- 导入后的城市内容统一挂在 `CityEnvironment` 根节点下，供主场景长期使用
- 新的 `Buildings` 根节点已被指定为障碍根，用于 A* 规划障碍识别与建筑代理碰撞体自动配置
- 原有简易建筑布局不再作为主展示内容，仅保留主场景备份用于回退
- 已清理不再使用的备选城市、美术试验资源与旧道路/天空盒/绿化包，当前仓库默认围绕 `SimpleCityPackage` 维持

已落地的场景辅助工具：

- `SimpleCitySceneImporter.cs`
  - 用于把 `SimpleCityPackage/Scene 01` 一键合并进主场景
  - 自动创建主场景备份
  - 自动更新 `DroneManager.obstacleRoot`
- `SimpleCityEnvironmentTools.cs`
  - 提供 `Apply Simple City Day Environment`
  - 提供 `Frame Overview Camera To City`
  - 用于把白天天空盒、雾效、太阳光和总览镜头恢复到更适合当前城市场景的状态

当前判断：

- 这一轮调整的目标不是继续“手搓城市”，而是把系统主线稳定建立在可直接复用的现成街区上
- 这样能把后续时间集中到无人机系统、算法接入、实验导出和答辩展示，而不是持续消耗在低收益的场景搭建上

## 6. 当前仍未真正完成或仍需继续加强的部分

### 6.1 多机冲突规避

当前情况：

- 项目已有多机调度和路径执行
- 已补上第一版简单规避策略
- 当前采用“近距离让行 + 高编号等待 + 冲突解除后自动恢复”的可解释方案
- 在当前实现下，仍能观察到部分近距离碰撞重叠与穿透现象，说明这个问题不能靠简单局部规则彻底解决

影响：

- 对近距离接近和目标区域重叠的场景，系统已具备基础等待/恢复行为
- 但还不是更强的路段占用、时空预约或全局协同方案
- 如果继续强行在现有规则上打补丁，收益可能有限，且会挤占系统搭建阶段的开发时间

当前决策：

- 该问题先记为“待专题研究”的算法项，不作为当前系统搭建阶段的阻塞任务
- 后续建议先补论文调研，再决定采用路径段占用、优先级规划、时空预约、CBS、ORCA 或其他更正式的方法

### 6.2 实时统计面板

仍可继续加强：

或
